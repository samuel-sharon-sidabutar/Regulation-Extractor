# code-style
- Break up large functions (like extract_main_content and main) into smaller, focused units. Confidence: 0.70
- Prefer guard clauses over long if-else chains for early returns and conditionals. Confidence: 0.80
- Use descriptive action-based function names (e.g., prune_malformed_rows) instead of underscore-prefixed private names (e.g., _handle_malformed_prune). Avoid generic names like run_phase1 that don't convey what the function actually does. Never use underscore prefixes on function names. Confidence: 0.85
- Follow Single Responsibility Principle: each function should do one thing and own that one thing completely — a function named for one task should not trigger unrelated side effects (e.g., print_batch_report should not call maybe_combine_csvs); let the caller sequence those steps. Confidence: 0.75
- Avoid naming functions with hedging words like "maybe" or "try" that encode runtime decisions — inline the conditional logic at the call site instead. Confidence: 0.75
- Replace generic action/type parameters with explicit boolean flags (e.g., move_pdf_on_failure instead of action param) to decouple functions from the caller's action system and make intent clear. Confidence: 0.70
- Functions that perform work should return their results so the caller can decide whether to proceed (e.g., convert_pdfs_to_markdown_batch should return the list of successfully converted files). Confidence: 0.70
- When a function takes many parameters (e.g., 7+), group related state into a small dict or dataclass rather than passing naked individual arguments. Confidence: 0.70
- Use full descriptive variable names instead of abbreviations (e.g., file_path instead of fp, base_name instead of b_name, result instead of r). Confidence: 0.70
- Prefer returning values from initialization/setup functions over mutating module-level globals. Confidence: 0.65
